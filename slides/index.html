What is a theorem prover?

  • A language in which you write both the theorem and the proof as code.
  • A kernel that mechanically checks the proof, term-by-term, against a small set of trusted rules.
  • If the kernel accepts it, the theorem is proved — with the same certainty as a verified program type-checks.

Key distinction from model checking:

  • Model checking: exhaustive search over a finite state space ⇒ yes / counterexample.
  • Theorem proving: symbolic reasoning, works over infinite state spaces / parametric systems, but requires human guidance.

A short family tree

  • LCF (1972, Milner) — the "trust the kernel" idea.
  • HOL / Isabelle/HOL — classical higher-order logic. Used for L4.verified (seL4 microkernel).
  • Coq / Rocq (1989) — Calculus of Inductive Constructions. Four Color Theorem, CompCert C compiler.
  • PVS (SRI, 1990s) — predicate subtypes, TCCs. Used in aerospace / NASA; the course's prior vehicle.
  • Agda, Idris — dependently-typed programming.
  • Lean 4 (Leo de Moura, 2013 → 2021) — this talk.

Curry–Howard: proofs ≡ programs

Logic side

  • Proposition P
  • Proof of P
  • P → Q
  • P ∧ Q
  • P ∨ Q
  • ∀ x, P x
  • ∃ x, P x

Programming side

  • Type P
  • Term of type P
  • Function P → Q
  • Pair (P, Q)
  • Tagged union
  • Dependent function
  • Dependent pair

Consequence: building a proof is the same activity as constructing a program that inhabits a type. The kernel "type-checks" both.

Lean 4 in one slide

  • Dependent type theory — Calculus of Inductive Constructions, tuned for usability.
  • Both a proof assistant and a practical programming language; Lean itself is largely written in Lean.
  • Tactic language for interactive proof: intro, apply, cases, induction, simp, omega, decide.
  • Mathlib — the world's largest formal math library (> 1.5M lines, > 200k theorems).
  • Macro system & elaborator let you extend the language itself.
  • Used in production: AWS (cryptographic code), Carnegie Mellon & Stanford courses, Terence Tao's polymath projects.

Why now? AI + math + formal proof

  • Terence Tao has been formalizing recent research in Lean (PFR conjecture, 2023) — in weeks instead of years.
  • DeepMind AlphaProof (2024): IMO silver-medal performance, output in Lean.
  • Harmonic (Marcus / Aaronson et al.) — "mathematical superintelligence" built on Lean.
  • Axiom Math / Math Inc. — research labs using neural proof search + Lean as ground truth.
  • Google's open Terry Tao-style projects mix LLM drafts with Lean verification.

The bet: LLMs hallucinate, but a Lean proof either type-checks or it doesn't. Formal methods become the oracle that grades the model's output.

Why a verification course should care

  • Model checking (NuXMV) answers finite, concrete safety questions.
  • Theorem proving (Lean) answers parametric, infinite, symbolic questions — and the same kind of questions, with proofs that compose.
  • The inductive invariant you write for NuXMV is exactly the invariant you'd prove in Lean.
  • When the state space blows up, or a property is parameterized over all n, theorem proving is the next step.

Part 1 — Discrete Math

Sets as predicates, basic proofs, tactic mode.

Sets as predicates


-- A set of α-elements is a function α → Prop
def MySet (α : Type) := α → Prop

def mem    (x : α) (A : MySet α)  : Prop := A x
def subset (A B : MySet α)        : Prop := ∀ x, mem x A → mem x B
def union  (A B : MySet α)        : MySet α := fun x => mem x A ∨ mem x B
def inter  (A B : MySet α)        : MySet α := fun x => mem x A ∧ mem x B
def compl  (A : MySet α)          : MySet α := fun x => ¬ mem x A

No axioms, no set theory baggage. Everything is a type or a function between types.

First proof: subset transitivity


theorem subset_trans (A B C : MySet α)
    (hab : subset A B) (hbc : subset B C) : subset A C := by
  intro x hxa
  apply hbc
  apply hab
  exact hxa
  • intro x hxa — suppose x and that x ∈ A.
  • apply hbc — reduce goal x ∈ C to x ∈ B.
  • apply hab — reduce goal x ∈ B to x ∈ A.
  • exact hxa — done.

Four tactics, each mechanical. This is the core rhythm of Lean.

De Morgan for sets


theorem demorgan_union (A B : MySet α) :
    seteq (compl (union A B)) (inter (compl A) (compl B)) := by
  intro x
  constructor
  · intro h
    constructor
    · intro ha; apply h; left; exact ha
    · intro hb; apply h; right; exact hb
  · intro ⟨hna, hnb⟩ hab
    cases hab with
    | inl ha => exact hna ha
    | inr hb => exact hnb hb

Part 2 — Transition Systems

The same structures we model-check in NuXMV, formalized.

A general transition system


structure TransitionSystem (State : Type) where
  init : State → Prop
  next : State → State → Prop

-- Reachable states defined inductively
inductive Reachable {State : Type} (ts : TransitionSystem State) : State → Prop
  | init (s : State) : ts.init s → Reachable ts s
  | step (s s' : State) : Reachable ts s → ts.next s s' → Reachable ts s'

Matches what we did in PVS — and what every model checking textbook calls "Post*" / "reach set".

Inductive invariants — the classical recipe


def InductiveInvariant (ts : TransitionSystem State) (P : State → Prop) : Prop :=
  (∀ s,    ts.init s   → P s) ∧
  (∀ s s', P s → ts.next s s' → P s')

-- Soundness: an inductive invariant holds on every reachable state.
theorem inductive_invariant_holds (ts : TransitionSystem State) (P : State → Prop)
    (h : InductiveInvariant ts P) : ∀ s, Reachable ts s → P s := by
  intro s hr
  induction hr with
  | init s hi         => exact h.1 s hi
  | step s s' _ hn ih => exact h.2 s s' ih hn

This single theorem is the backbone of everything that follows. Once we prove an invariant is inductive, reachability gives us the universal property for free.

Two useful corollaries


-- Invariant strengthening: if P is inductive and P → Q, then Q is invariant.
theorem invariant_strengthening (ts : _) (P Q : State → Prop)
    (hind : InductiveInvariant ts P) (himp : ∀ s, P s → Q s) :
    ∀ s, Reachable ts s → Q s := ...

-- Conjunction: P ∧ Q inductive ⇐ P inductive ∧ Q inductive.
theorem invariant_conjunction (ts : _) (P Q : _) ... := ...

Why strengthening matters: the property we want is often not inductive. We prove a stronger property (the "true invariant") and then weaken it.

Part 3 — NuXMV → Lean

Parse .smv files with a grammar, generate Lean, prove invariants.

The counter model

counter.smv


MODULE main
  VAR
    mode : {off,on};
    press : boolean;
    x : 0..25;
  DEFINE count_max := 10;
  ASSIGN
    init(mode) := off;
    init(x)    := 0;
    next(mode) := case
      mode=off & !press : off;
      mode=off &  press : on;
      mode=on & !press & x=count_max) : off;
      TRUE : mode;
    esac;
    next(x) := case
      mode=on & !press & x=count_max) : 0;
      TRUE : x;
    esac;
  INVARSPEC x <= count_max;
  INVARSPEC (mode = off) -> (x = 0);

Pipeline

  1. Lark LALR(1) grammar parses the .smv file.
  2. Transformer builds an SmvModel IR.
  3. smv2lean.py emits a Lean file.

The same generator is what we reuse from the smvis project.

Generated Lean (snippet)


inductive ModeVal where | off | on deriving DecidableEq, Repr
open ModeVal

structure CounterState where
  mode : ModeVal
  press : Bool
  x : Nat
  deriving DecidableEq, Repr

def CounterTS : TransitionSystem CounterState where
  init s := s.mode = .off ∧ s.x = 0
  next s s' :=
    ∃ press' : Bool,
    s'.press = press' ∧
    (if ((s.mode = .off) ∧ (s.press = false)) then s'.mode = .off
    else if ((s.mode = .off) ∧ (s.press = true))  then s'.mode = .on
    else ...) ∧
    (if (((s.mode = .on) ∧ (s.press = false)) ∧ (s.x < 10)) then s'.x = s.x + 1
    else if ((s.mode = .on) ∧ ((s.press = true) ∨ (s.x ≥ 10))) then s'.x = 0
    else s'.x = s.x)

press has no init/next ⇒ detected as a nondeterministic input, existentially quantified at each step.

Proving the invariants — strengthen first

The two INVARSPECs we want: x ≤ 10 and mode=off → x=0. Neither is inductive on its own. The conjunction is.


def counterInv (s : CounterState) : Prop :=
  s.x ≤ 10 ∧ (s.mode = .off → s.x = 0)

theorem counterInv_inductive : InductiveInvariant CounterTS counterInv :=
  ⟨counterInv_init, counterInv_step⟩   -- each a short case analysis

-- Individual INVARSPECs follow by strengthening.
theorem CounterTS_inv1_proved :
    Invariant CounterTS (fun s => s.x ≤ 10) :=
  invariant_strengthening CounterTS counterInv _ counterInv_inductive
    (fun _ h => h.1)

theorem CounterTS_inv2_proved :
    Invariant CounterTS (fun s => s.mode = .off → s.x = 0) :=
  invariant_strengthening CounterTS counterInv _ counterInv_inductive
    (fun _ h => h.2)

Euclid's GCD — a deeper example

Algorithm


l1: while (a ≠ b) {
l2:   if (a > b)
l3:     a := a - b
      else
l4:     b := b - a
l5: }

Encoding: a program counter pc ∈ {l1,…,l5}. Transitions update pc, a, or b based on the current line.

What we prove

  1. a ≥ 0 ∧ b ≥ 0 — trivially (Nat).
  2. pc = l5 → a = b — termination correctness, by induction over transitions.
  3. gcd(a, b) is preserved by every step — the classical correctness result.

Each fact uses the same inductive_invariant_holds theorem, specialized.

GCD preservation — the core lemma


-- The identity underlying Euclid's algorithm, available in core Lean 4:
theorem gcd_sub_eq (a b : Nat) (h : b ≤ a) :
    Nat.gcd (a - b) b = Nat.gcd a b :=
  Nat.gcd_sub_self_left h

-- One-step GCD preservation.
theorem GcdTS_gcd_preserved :
    ∀ s s', GcdTS.next s s' → Nat.gcd s.a s.b = Nat.gcd s'.a s'.b := by
  intro s s' ⟨ha_next, hb_next, _⟩
  by_cases h3 : s.pc = .l3 ∧ s.a > s.b
  · -- a := a - b, b unchanged → gcd invariant by gcd_sub_eq
    ...
  · by_cases h4 : s.pc = .l4 ∧ s.b > s.a
    · -- symmetric, uses Nat.gcd_comm
      ...
    · -- neither changes; done.
      ...

Proof structure: case-split on which line of code we are at, reuse the stdlib GCD identity.

Mutex — 2 processes (NuXMV model)


-- Strengthened inductive invariant for Peterson-style mutex:
def mutexInv (s : MutexState) : Prop :=
  (p1InLock s ↔ s.flag1 = true) ∧
  (p2InLock s ↔ s.flag2 = true) ∧
  ¬(s.process1 = .critical ∧ s.process2 = .critical)

theorem MutexTS_mutual_exclusion :
    Invariant MutexTS (fun s => ¬(s.process1 = .critical
                                  ∧ s.process2 = .critical)) :=
  invariant_strengthening MutexTS mutexInv _
    mutexInv_inductive (fun _ h => h.2.2)

Key insight: mutual exclusion alone is not inductive. We strengthen with flag-coupling invariants. Then: for both to enter `critical` in one step, both must be `waiting` ⇒ both flags true ⇒ entry conditions force turn = 1 ∧ turn = 2 ⇒ contradiction.

Parametric mutex — all N processes

This is what NuXMV cannot do: prove mutual exclusion for all N ∈ ℕ simultaneously.


structure MState (n : Nat) where
  procs  : Fin n → ProcState       -- n process states
  locked : Bool                    -- the shared lock

def mutexTS (n : Nat) : TransitionSystem (MState n) where
  init s := s.locked = false ∧ ∀ i, s.procs i = .ncs
  next s s' := ∃ i : Fin n,
    (∀ j, j ≠ i → s'.procs j = s.procs j) ∧              -- frame
    (⟨acquire i⟩ ∨ ⟨release i⟩ ∨ ⟨stutter⟩)

def atMostOneCS (s : MState n) : Prop :=
  ∀ i j, s.procs i = .cs → s.procs j = .cs → i = j

theorem mutual_exclusion (n : Nat) :
    Invariant (mutexTS n) (fun s => atMostOneCS s) := ...

One proof, universally quantified over n. Model checking N=2, N=3, … ad infinitum is replaced by a single inductive argument.

Why this composition matters

  • The same inductive-invariant machinery we built once works for every model the parser emits.
  • Different .smv files → different State types, same proof strategy.
  • Tested on: counter.smv, gcd.smv, traffic_light.smv, mutex.smv.
  • Compare to NuXMV: model-check passes/fails on specific bounded instances; Lean gives you the parametric result, plus a reusable proof you can extend.
  • Parametric verification — proving a property for all N at once — is the defining capability of theorem proving that model checking genuinely cannot match.

Part 4 — Program Verification

IMP language, big-step semantics, Hoare logic — loop invariants in Lean.

A tiny Python-flavored language


inductive AExp | num : Int → AExp | var : String → AExp
             | add | sub | mul : AExp → AExp → AExp

inductive BExp | tt | ff | eq | le | lt | not | and (…)

inductive Cmd
  | skip
  | assign : String → AExp → Cmd
  | seq    : Cmd → Cmd → Cmd
  | ifte   : BExp → Cmd → Cmd → Cmd
  | while  : BExp → Cmd → Cmd

-- Big-step operational semantics: BigStep c s s' means running c in s reaches s'
inductive BigStep : Cmd → PState → PState → Prop where
  | skip   : BigStep .skip s s
  | assign : BigStep (.assign x e) s (s[x ↦ aeval e s])
  | seq    : BigStep c1 s s' → BigStep c2 s' s'' → BigStep (.seq c1 c2) s s''
  | while_false : beval b s = false → BigStep (.while b c) s s
  | while_true  : beval b s = true → BigStep c s s' → BigStep (.while b c) s' s''
                → BigStep (.while b c) s s''

Hoare logic — the six standard rules


def HoareTriple (P : Assertion) (c : Cmd) (Q : Assertion) : Prop :=
  ∀ s s', P s → BigStep c s s' → Q s'

notation:50 "⦃" P "⦄ " c " ⦃" Q "⦄" => HoareTriple P c Q

theorem hoare_skip    : ⦃P⦄ skip ⦃P⦄
theorem hoare_assign  : ⦃fun s => Q (s[x ↦ aeval e s])⦄ (x ::= e) ⦃Q⦄
theorem hoare_seq     : ⦃P⦄ c1 ⦃Q⦄ → ⦃Q⦄ c2 ⦃R⦄ → ⦃P⦄ (c1 ;; c2) ⦃R⦄
theorem hoare_if      : ... → ⦃P⦄ (if b then c1 else c2) ⦃Q⦄
theorem hoare_while   : ⦃I ∧ b⦄ c ⦃I⦄ → ⦃I⦄ (while b do c) ⦃I ∧ ¬b⦄
theorem hoare_consequence : strengthen precond / weaken postcond

All six proved from the big-step semantics — no axioms.

def swap : Cmd :=
  "t" ::= AExp.var "x" ;;
  "x" ::= AExp.var "y" ;;
  "y" ::= AExp.var "t"

theorem swap_correct (a b : Int) :
  ⦃fun s => s "x" = a ∧ s "y" = b⦄
    swap
  ⦃fun s => s "x" = b ∧ s "y" = a⦄

Loop invariant: 0 ≤ i ≤ n ∧ n = n₀. Proof uses hoare_while + omega.

Testing vs Verification

A sorting algorithm: the cleanest illustration.

Insertion sort — three views, one algorithm

Python (mutable list)


def sort(a):
    for i in range(1, len(a)):
        key = a[i]
        j = i - 1
        while j >= 0 and a[j] > key:
            a[j+1] = a[j]
            j = j - 1
        a[j+1] = key

Lean (imperative — same syntax!)


def sortArr (input : Array Int) := Id.run do
  let mut a := input
  for i in [1 : a.size] do
    let key := a[i]!
    let mut j : Nat := i
    while j > 0 ∧ a[j-1]! > key do
      a := a.set! j a[j-1]!
      j := j - 1
    a := a.set! j key
  return a

Lean (recursive — easier to prove)


def ins (x : Int) : List Int → List Int
  | []      => [x]
  | y :: ys => if x ≤ y then x :: y :: ys
               else y :: ins x ys

def sort : List Int → List Int
  | []      => []
  | x :: xs => ins x (sort xs)

Lean has full imperative syntax — let mut, for, while, in-place updates. The recursive form is just easier for the kernel to reason about. Both compute the same answer.

Testing — one input at a time


#eval sort [3, 1, 4, 1, 5, 9, 2, 6]   -- [1, 1, 2, 3, 4, 5, 6, 9]
#eval sort [5, 4, 3, 2, 1]            -- [1, 2, 3, 4, 5]
#eval sort ([] : List Int)            -- []
#eval sort [-3, 0, -1, 5, -10]        -- [-10, -3, -1, 0, 5]

-- Machine-checked tests — each is a theorem about ONE input.
example : sort [3, 1, 4, 1, 5, 9, 2, 6] = [1, 1, 2, 3, 4, 5, 6, 9] := by decide
example : sort [5, 4, 3, 2, 1]          = [1, 2, 3, 4, 5]          := by decide
example : sort ([] : List Int)          = []                       := by decide
example : sort [-3, 0, -1, 5, -10]      = [-10, -3, -1, 0, 5]      := by decide

Ten tests cover ten inputs. List Int has infinitely many values. Tests cannot cover them all — no matter how many you write.

Verification — one theorem, all inputs


-- Specification: what does it mean to be sorted?
inductive Sorted : List Int → Prop where
  | nil       : Sorted []
  | singleton : ∀ x, Sorted [x]
  | cons      : ∀ x y ys, x ≤ y → Sorted (y :: ys) → Sorted (x :: y :: ys)

-- ★ Correctness: for EVERY list, sort produces a sorted permutation.
theorem sort_correct : ∀ xs : List Int,
    Sorted (sort xs) ∧ List.Perm (sort xs) xs :=
  fun xs => ⟨sort_sorted xs, sort_perm xs⟩
  • Sortedness — output is in non-decreasing order.
  • Permutation — output has exactly the same elements (no element added, lost, or duplicated). Uses List.Perm from core Lean.
  • Together: "sort produces a sorted rearrangement" — and the ∀ xs means for every possible input.

Why this matters

  • You will never write enough tests to cover all List Int — the input space is infinite.
  • One Lean theorem, type-checked once, replaces an unbounded test suite.
  • If you change sort tomorrow and it breaks correctness, the proof fails to compile. The compiler is your regression test.
  • This is the same pattern from earlier: an inductive invariant for the recursion (here, ins_sorted) underpins the global theorem.

The same idea, three times

  • Transition systems: inductive invariant P. Prove: init → P, P ∧ step → P'.
  • NuXMV models: same, plus pc to track control flow.
  • Hoare logic: loop invariant I. Prove: pre → I, I ∧ guard → I (body), I ∧ ¬guard → post.

The "inductive invariant" is one idea, wearing different clothes.

Sorting a list — the cleanest example of the difference.

Sorting — three views, one algorithm

Python (imperative)


def sort(a):
  for i in range(1, len(a)):
    key = a[i]
    j = i - 1
    while j >= 0 and a[j] > key:
      a[j+1] = a[j]
      j = j - 1
    a[j+1] = key

Our IMP (extended w/ arrays)


i := 1;
while i < len(a):
  key := a[i];
  j := i - 1;
  while j ≥ 0 && a[j] > key do
    a[j+1] := a[j];
    j := j - 1
  end;
  a[j+1] := key;
  i := i + 1

Lean (recursion = loops)


def ins (x : Int) : List Int → List Int
  | []      => [x]
  | y :: ys =>
      if x ≤ y
      then x :: y :: ys
      else y :: ins x ys

def sort : List Int → List Int
  | []      => []
  | x :: xs => ins x (sort xs)

Same algorithm, three syntaxes. The proof technology — loop / recursion invariants — is identical.

Testing — what specific cases tell us


#eval sort [3, 1, 4, 1, 5, 9, 2, 6]   -- [1, 1, 2, 3, 4, 5, 6, 9]
#eval sort [5, 4, 3, 2, 1]            -- [1, 2, 3, 4, 5]
#eval sort ([] : List Int)            -- []
#eval sort [-3, 0, -1, 5, -10]        -- [-10, -3, -1, 0, 5]

-- Each test is a theorem about ONE specific input:
example : sort [3, 1, 4, 1, 5, 9, 2, 6] = [1, 1, 2, 3, 4, 5, 6, 9] := by decide
example : sort [5, 4, 3, 2, 1]          = [1, 2, 3, 4, 5]          := by decide
example : sort ([] : List Int)          = []                       := by decide

Ten tests cover ten inputs. List Int has infinitely many values. Tests can never cover them all. They tell us what the function does for the inputs we wrote — and nothing about anything else.

The correctness criterion — what does "sorted" even mean?


-- A list is sorted iff every adjacent pair is in non-decreasing order.
inductive Sorted : List Int → Prop where
  | nil       : Sorted []
  | singleton : ∀ x, Sorted [x]
  | cons      : ∀ x y ys, x ≤ y → Sorted (y :: ys) → Sorted (x :: y :: ys)

Two parts to "correct":

  1. Sortedness — the output is in non-decreasing order.
  2. Permutation — the output contains exactly the same elements as the input (no elements added, lost, or duplicated).

Without (2), a "sort" function that always returned [] would satisfy (1).

Verification — one theorem, all inputs


-- Property (1): the output is sorted, FOR EVERY list xs.
theorem sort_sorted : ∀ xs : List Int, Sorted (sort xs)

-- Property (2): the output is a permutation of the input, FOR EVERY xs.
theorem sort_perm   : ∀ xs : List Int, List.Perm (sort xs) xs

-- Combined: SORT IS CORRECT FOR ALL LISTS OF INTEGERS.
theorem sort_correct : ∀ xs : List Int,
    Sorted (sort xs) ∧ List.Perm (sort xs) xs :=
  fun xs => ⟨sort_sorted xs, sort_perm xs⟩

Read it: "for all xs in List Int — every list, of every length, every combination of integers — sort xs is sorted, and a permutation of xs."

One theorem covers what no finite test suite ever could.

The contrast in one slide

Testing Verification
Pick inputs, run, check output. State a property, prove it for all inputs.
Each test = one input. One theorem = infinitely many inputs.
Misses bugs in inputs you didn't try (Heartbleed, integer overflows, edge cases). Closed under quantification; if proved, no input can violate.
Cheap, fast, works on any system. Requires effort up front; output is a machine-checked guarantee.
"It works on my examples." "It works, period."

Takeaways

  • Lean 4 makes proofs feel like programming — same editor, same mental model.
  • Everything in this course that can be stated parametrically, can be stated and proved in Lean.
  • When a model becomes too large for NuXMV (or parametric, or infinite), theorem proving is the next rung on the ladder.
  • The AI+proof frontier is real, fast-moving, and Lean is its lingua franca.
